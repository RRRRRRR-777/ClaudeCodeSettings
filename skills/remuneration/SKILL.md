## 申請書記入支援

ユーザーから評価項目と参照先の提示を受け、以下の手順で `Level_3/20250501_rikuyamada.md` に記入する。

### ファイルパス

- 申請書: `Level_3/20250501_rikuyamada.md`（リポジトリルートから）
- 評価基準: `Level_3/self-scoring.md`（リポジトリルートから）

### 実行手順

**1. 入力の解析**

ユーザー入力から以下を特定する：
- 評価項目コード（例: 2-A-2）
- 参照先（GitHub/GitLab/Redmine URL など）
- 説明・概要（あれば）

**2. セクションの特定**

下記の対応表から記入先セクション（3.X）を特定する。

**3. 評価ポイントの確認**

Grep ツールで `self-scoring.md` を検索（`[LVX][X-X-X]` 形式）し、Good/Bad/解説を確認する。

**4. 既存フォーマットの確認**

Grep ツールで `20250501_rikuyamada.md` の該当セクション（`^### 3\.X`）を読み込み、既存の記載フォーマットを確認する。

**5. 不明点の質問（必須）**

以下の場合は必ず選択式で質問する（推測で記入しない）：
- コードが特定できない
- セクションが複数該当する
- 概要の書き方に迷う
- フォーマットの解釈が曖昧

**6. Jiraエピックチケットにコメント追記（必須・確認なし）**

ユーザー確認なしで**必ず実行**する。**1項目1コメント**で投稿する。

チケットIDは以下の優先順で特定する：
1. 会話履歴に登場したチケットID（例: KAN-XX）
2. ユーザーが貼ったJira/URL
3. 上記で判断できない場合のみ「どのエピックチケットに記録しますか？」と質問

**コメントフォーマット**:
- ヘッダーは**項目名のみ**をH1で記載する（例: `# エラーハンドリング`）
- `[報酬制度チェック]` プレフィックスや評価コード（2-A-2等）、申請書セクション番号はタイトルに含めない
- **表形式で網羅的・詳細に記載する**（異常系一覧表、制約一覧表、アクセスパターン分析表等）
- 箇条書き2〜3行の簡潔な記載ではなく、根拠・テスト記録・ファイルパス等を含む十分な詳細を書く
- 「申請書に記入しますか？」の確認は不要

**7. 申請書への記入**

記入ルール：
- 重複するURLは除外する
- 新しい項目は既存項目の**上から**追加する
- 追加後に番号を1から振り直す
- 既存フォーマットに従う

**8. 完了報告**

```
記入完了しました

【記入内容】
- 評価項目: [コード] / [項目名]
- セクション: 3.X
- 追加件数: X件
- 重複スキップ: X件

【Jiraコメント追記】
- チケット: [推測/入力されたID]
- 追記内容: 評価コード・セクション・参照先・記入日時

【評価ポイント参考】
- Good: [self-scoring.md より]
- 注意点: [self-scoring.md より]

ファイルを確認して、必要に応じて調整してください。
```

### 注意事項

- 両ファイルは大きいため Grep のみで必要箇所を取得する（全体読み込み不可）
- URLだけでなく、評価ポイントを意識した概要・説明を必ず付ける
