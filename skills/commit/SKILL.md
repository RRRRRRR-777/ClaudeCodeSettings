# Git Commit Skill

## 目的

このSkillは、AIが一貫性・可読性・再現性のあるGitコミットを行うための最小かつ汎用的なルールを定義する。

- 現在の差分のみを根拠にコミットする
- **過去のコミットスタイルを参照してルールを決めない（skillで定義されたルールに従う）**
- 人間・AIの双方にとって読みやすい履歴を残す
- **コミット粒度は細かく保つ**

---

## 実行手順

このスキルが呼び出されたとき、以下の手順を実行する。
「コミットしますか？」の質問は不要です。

### Step 1. 事前確認（必須）
```bash
git status
git diff --staged
git diff
```

- `git status` を実行し、現在の変更差分のみを確認する
- **`git log` を実行して過去のコミットスタイルを調べない（現在の差分のみで判断する）**

### Step 2. コミット対象の判定 → パス分岐

ここで状態に応じて **1つだけパスを選び、そのパスの手順を最後まで実行する**。
他のパスの手順には進まない。

```
引数でファイル指定あり？ → [パスC] へ
ステージング済みファイルあり？ → [パスA] へ
どちらでもない → [パスB] へ
```

---

## パスA: ステージング済みファイルがある場合

ステージング済みファイルはユーザーがレビュー・選別済みの証。
このパスでは **ステージング済みの変更だけをコミットし、未ステージングの変更には一切触れない**。

### A-1. 粒度チェック（必須）

ステージング済みであっても、粒度の判断はスキップしない。
`git diff --staged` の内容を分析し、**パスBと同じ分割基準**で複数の論理単位が混在していないか確認する。

**混在の判断観点**:
- 異なる機能/モジュールのファイルが含まれていないか（例: スキルAとスキルBの変更）
- 異なる目的の変更が含まれていないか（例: 新規作成 + 既存修正 + 設定変更）
- 1文で「このコミットは何をしたか」を説明できるか

**混在している場合** → 一度unstageし、論理単位ごとにステージング→コミットを繰り返す（パスBのB-2と同じ手順）。

```bash
# unstage
git reset HEAD -- [ファイル群]

# 論理単位ごとに再ステージング → コミット
git add [論理単位1のファイル群]
git commit -m "..."

git add [論理単位2のファイル群]
git commit -m "..."
```

**混在していない場合** → A-2へ進む。

### A-2. コミットメッセージの作成と実行

`git diff --staged` の内容だけを根拠に、コミットメッセージ形式（後述）に従ってコミットする。

```bash
git commit -m "$(cat <<'EOF'
<type>: <title> (#<チケット番号>)

- 変更内容1
- 変更内容2
EOF
)"
```

### A-3. Push確認 → 終了

```
コミットが完了しました。リモートにPushしますか？ (yes/no):
```

- ユーザーが「はい / yes / y」と回答した場合のみ `git push` を実行する
- それ以外の回答、または無回答の場合は Push しない

**ここでスキルの実行は完了。** 未ステージングの変更が残っていても触れない。

---

## パスB: ステージング済みファイルがない場合

未ステージングの変更を分析し、**最小の論理単位**で分割してコミットする。

### B-1. 分割の判断

**分割の判断基準** — 以下のいずれかに該当する場合、**必ず**複数コミットに分割する：
- 異なるtypeの変更が混在している（例: 機能追加 + バグ修正）
- 異なる機能/モジュールの変更が混在している
- テストコードと実装コードが別の論理単位に属する場合
- 設定ファイルの変更と機能コードの変更が混在している

**分割の粒度目安**:
- 1コミット = 1つの論理変更（1ファイル〜数ファイル程度）
- 「このコミットは何をしたか」を1文で説明できる単位
- テストと実装が同じ機能に属する場合はまとめてよい

**ユーザーに分割の確認は不要。自動で最適な粒度に分割して実行する。**

### B-2. ステージング → コミットを繰り返す

論理単位ごとに以下を繰り返す：

```bash
# 1. 論理単位ごとに特定ファイルをステージング
git add [対象ファイル1] [対象ファイル2]

# 2. コミット
git commit -m "$(cat <<'EOF'
<type>: <title> (#<チケット番号>)

- 変更内容1
- 変更内容2
EOF
)"
```

**注意**: `git add .` は粒度が荒くなるため原則使用しない。論理単位ごとにファイルを指定する。

全ての論理単位が完了するまで繰り返す。

### B-3. Push確認 → 終了

```
コミットが完了しました。リモートにPushしますか？ (yes/no):
```

- ユーザーが「はい / yes / y」と回答した場合のみ `git push` を実行する
- それ以外の回答、または無回答の場合は Push しない

---

## パスC: 引数でファイルが指定されている場合

- 指定ファイルのみをステージング → コミットする（ステージング状態に関わらず）
- 除外ファイルが明示されている場合は、指定ファイルをコミット対象から除外する
- コミット後、Push確認 → 終了（パスA/Bと同じ）

---

## チケット番号について
- ブランチ名が `ticket/{番号}` の場合、その番号を使用する
- チケット番号が不明な場合は省略してよい

---

## 基本原則

- **1コミット = 1つの最小論理変更**
- コミットは小さく、意味的に完結させる
- コミットメッセージは「何を」「なぜ」を伝える
- 推測や差分に存在しない内容は書かない
- **粒度が荒いと感じたら迷わず分割する**

---

## コミットメッセージ形式（固定）

```
<type>: <title> (#<チケット番号>)

- 変更内容1
- 変更内容2
- 変更理由 / 影響（必要な場合のみ）
```

### ルール
- **scopeは使用しない**（`type(scope):` ではなく `type:` を使う）
- チケット番号はタイトル末尾にスペース+括弧で付与する
- タイトルと本文の間に空行を1行入れる
- 本文は3行目以降・箇条書きのみ
- 本文は必要最小限に留める
- **必ず日本語で書く**

---

## プレフィックス（type）

| type | 用途 |
|------|------|
| feat | 機能追加 |
| fix | バグ修正 |
| refactor | 挙動を変えない改善 |
| docs | ドキュメント変更 |
| test | テスト追加・修正 |
| chore | 設定・ビルド・雑務 |

---

## タイトルの書き方

- 50文字以内を目安
- 命令形・現在形で書く（例: 「追加した」→「追加」）
- 実装手段ではなく変更結果を書く
- 句点（。）は付けない
- **必ず日本語で書く**

---

## 本文の書き方

- 箇条書きのみを使用する
- diff から読み取れる事実のみを書く
- 「なぜ変更したか」が重要な場合のみ理由を書く
- 行数・文字数は最小限
- **必ず日本語で書く**

---

## コミット粒度ガイドライン

### 細かく分割すべきケース
| パターン | 分割方法 |
|---------|---------|
| 新規ファイル追加 + 既存ファイル修正 | 新規追加と既存修正を分ける |
| BE変更 + FE変更 | バックエンドとフロントエンドを分ける |
| 実装 + テスト（別機能） | 機能ごとにまとめる |
| マイグレーション + コード変更 | マイグレーションを先にコミット |
| 設定変更 + ロジック変更 | 設定と実装を分ける |
| ドキュメント + コード | ドキュメントとコードを分ける |
| **クリーンアーキテクチャの複数層** | **層ごとに分割する（下記参照）** |

### 目的

- アーキテクチャ非依存。依存の向きに沿って、抽象→具体でコミット。

### 基本原則

- 依存方向優先: 提供者→利用者の順で積む。
- 契約先行: スキーマ/インターフェース/DTO/イベントを先に固定。
- 境界単位: 境界ごとにまとめ、同境界テストを同梱。
- 常時安定: 各コミットはビルド/テストが通る状態。

### コミット順序（依存方向順）

- 契約・スキーマ
- コアロジック（ビジネスルール）
- 外部接続点（ゲートウェイ/クライアント）
- データアクセス（リポジトリ/ORM）
- 調停（ユースケース/サービス）
- 入出力・公開面（ハンドラ/コントローラ/ルータ/設定）
- 既存追随修正（型変更・移行対応）

### 適用条件

- 1機能が3境界以上に跨る場合に適用。
- 1〜2境界の変更は通常粒度で判断。

### まとめてよいケース

- 1機能の実装＋同機能のテスト。
- 分割で壊れる密結合の同一変更。

### 例外（小規模スライス）

- 小さな変更で「契約→実装→配線」が少数ファイルに収まる場合は1コミットでも可（常に緑を維持）。

---

## 禁止事項

- scope記法の使用（`type(scope):` は禁止、`type:` のみ）
- 複数typeの併用
- 差分に存在しない変更の記載
- 感情的・主観的な表現
- 過去コミットとの比較・言及
- **英語でのコミットメッセージ**
- **`git log` を参照してコミットスタイルを決めること**
- **`git add .` による一括ステージング（粒度が荒くなるため）**

---

## 例

### 良い例

```
feat: MFAログインを追加 (#163)

- TOTPベースの多要素認証をログインフローに追加
- 検証コード入力UIを追加
```

```
fix: ユーザー取得時のヌルポインタを修正 (#163)

- プロフィールデータ欠落時のガード処理を追加
- ヌルシナリオの単体テストを追加
```

```
docs: API仕様書を更新 (#163)
```

### 悪い例

```
docs(commit): スキルを更新   ← scope記法は禁止
```

```
fix stuff   ← 英語は禁止
```

```
feat: ログイン機能追加、DB設計変更、テスト追加   ← 粒度が荒い、分割すべき
```

---

## このSkillの一文定義

現在の差分だけを根拠に、最小論理単位で分割し、定型フォーマットで迷いなくコミットを作る
