# Rule Creator

## 目的

ユーザーが表明した振る舞いの追加・修正を、適切な形式と場所に保存する。
「ルールを追加して」「〜と言われたら〜するようにして」などの要望を受けて起動する。

---

## 実行手順

### 1. 既存 rules を検索

以下の2箇所をすべてスキャンする（Glob ツールを使用）:
- `~/.claude/rules/` — ユーザーレベル（全プロジェクト共通）
- `.claude/rules/` — プロジェクトレベル（現在の作業ディレクトリ）

ファイルが存在する場合は内容を読み込み、ユーザーの要求と関連するものを探す。

### 2. 既存ルールの修正 or 新規作成

**関連ファイルが見つかった場合**: 既存ファイルの該当箇所を修正・改善して上書きする
**関連ファイルがない場合**: rule vs skill の判断基準（下記）に従い、保存形式を決定する

---

### 3. rule vs skill の判断基準

| 条件 | 保存形式 |
|------|---------|
| 1〜3行の宣言文で収まる。「常に〜せよ」「〜は禁止」などの単純な制約 | **rule** として保存 |
| 手順・条件分岐・複数ステップを含む。500トークン以上必要になる場合 | **skill** として切り出す |

### 4. 保存先の決定

**rule の場合**: AskUserQuestion でスコープを確認する
- ユーザースコープ（全プロジェクト共通） → `~/.claude/rules/[name].md`
- プロジェクトスコープ（現在のプロジェクトのみ） → `.claude/rules/[name].md`

**skill の場合**:
- `~/.claude/skills/[name]/SKILL.md` に作成する（ユーザーレベル固定）

### 5. ファイル名の決定

ユーザーの要求内容から簡潔な英語スネークケース名を自動生成する。
例: 「回答は箇条書きで」→ `answer_format.md`

---

### 6. 確認フォーマット

保存前に必ずユーザーに提示し、確認を取ってから Write / Edit を実行する。
操作の種類（新規作成・修正・skill化）によってフォーマットを変える。

#### 6-A: rule を新規作成する場合

```
📄 新規 rule を作成します

保存先 : ~/.claude/rules/answer_format.md
スコープ: ユーザーレベル（全プロジェクトに適用）
トリガー: 常時（全レスポンスに適用）

─────────────────────────────
回答は常に箇条書き形式で記述すること。
長文の段落は使用しない。
─────────────────────────────

この内容で保存しますか？
```

#### 6-B: 既存 rule を修正する場合

```
✏️ 既存 rule を修正します

対象ファイル: ~/.claude/rules/answer_format.md

─── 変更前 ──────────────────
回答は箇条書き形式で記述すること。
─────────────────────────────

─── 変更後 ──────────────────
回答は常に箇条書き形式で記述すること。
長文の段落は使用しない。
コードは必ずコードブロックで囲む。
─────────────────────────────

変更内容: 「常に」を追加、段落禁止とコードブロック要件を追記

この変更を適用しますか？
```

#### 6-C: skill として新規作成する場合

```
🛠️ 複雑なフローのため Skill として作成します

保存先: ~/.claude/skills/pr-review/SKILL.md
理由  : 手順・条件分岐を含むため rule には収まらない

Skill の概要:
  - PR のタイトル・差分・コメントを確認
  - セキュリティ・パフォーマンス・可読性を評価
  - フォーマット済みレビューコメントを出力

この内容で Skill を作成しますか？
（「はい」の後に SKILL.md の詳細を生成します）
```

確認後、Write または Edit ツールでファイルを保存する。

---

## 使用例

### 例1: シンプルなルールの追加

```
User: /rule-creator 回答は常に箇条書きで

→ ~/.claude/rules/ を検索 → 該当なし
→ rule と判断（1行で収まる宣言文）
→ スコープ確認（AskUserQuestion）
→ 確認フォーマット 6-A を提示
→ 承認後: ~/.claude/rules/answer_format.md を作成
```

### 例2: 既存ルールに関連する変更

```
User: /rule-creator コードは必ずコードブロックで囲んで

→ ~/.claude/rules/answer_format.md が見つかる
→ 既存ファイルに追記と判断
→ 確認フォーマット 6-B（diff形式）を提示
→ 承認後: Edit ツールで既存ファイルを更新
```

### 例3: フローを伴うルール → Skill 化

```
User: /rule-creator PR レビューするときは差分確認・セキュリティチェック・コメント生成の順でやって

→ ~/.claude/rules/ と .claude/rules/ を検索 → 該当なし
→ 手順・分岐あり → skill と判断
→ 確認フォーマット 6-C を提示
→ 承認後: ~/.claude/skills/pr-review/SKILL.md を生成
```

---

## 注意事項

- rules ディレクトリが存在しない場合は `mkdir -p` で作成してから保存する
- 既存ファイルの編集は Edit ツールを使用する（ファイル全体の上書きは避ける）
- skill の SKILL.md は commit/SKILL.md 等と同じ YAML フロントマター形式で作成する
- プロジェクトスコープを選択した場合、`.claude/rules/` ディレクトリがなければ作成する

---

## このSkillの一文定義

振る舞いの要求を受け取り、rule か skill かを判断して適切な場所に保存する
